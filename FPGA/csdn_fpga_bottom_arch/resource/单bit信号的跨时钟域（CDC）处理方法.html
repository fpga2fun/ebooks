0、前言

        跨时钟域的处理可以分为两个大类：单Bit信号跨时钟域处理、多Bit信号跨时钟域处理。分类的原因是多bit信号的传递不光只有亚稳态这一个问题，还可能会因为多个信号之间由于工艺、PCB布局等因素导致的信号传输延时（skew）的存在，从而导致信号被漏采或者错采。

        既然信号的跨时钟域可能会引入亚稳态问题，那么就需要想办法对其进行处理，从而降低亚稳态发生的概率（即提高MTBF）。

1、单bit信号的分类

        在FPGA设计中，单bit信号常常被分为两类：

（1）电平信号：

        多个时钟周期内都有效（通常为高电平有效）的信号，常常用来指示某个状态。比如UART发送时可以把tx_en拉高，表明当前正处于发送状态。

（2）脉冲信号： 

        单个时钟周期有效（通常为高电平有效）的信号，常常用来指示某个数据有效。比如data和data_valid配合使用，data_valid拉高即表明当前的数据data有效。

2、单bit信号进行跨时钟域处理的方法

        信号从源时钟域跨到目的时钟域后，需要在目的时钟的 “指挥” 下进行操作，这一过程可视为用目的时钟对被处理后的信号进行 “采集” ，既然是采集，那么就必须满足奈奎斯特采样定理，也就是说目的时钟频率需要至少是源时钟频率的2倍。

        在设计实践中，不光需要将慢时钟域信号同步到快时钟域，也需要将快时钟域信号同步到慢时钟域。前者可能无法满足采样定理，而后者则肯定无法满足采样定理，因此针对这些特殊情况需要一些特殊的处理办法。

2.1、电平信号的跨时钟域处理（电平同步器）

        因为电平信号通常在多个时钟周期内都有效，因此一般都不存在漏采的问题。当然了，如果目的时钟比源时钟的频率小了特别多，那仍然是有可能出现漏采的，这时的处理办法就和下面要说的快时钟域信号到慢时钟域的采集方法一致。

        实践中更常见的是目的时钟域可以保证采集到源时钟域的电平信号（不管是快到慢，还是慢到快），这时最常用的跨时钟域处理方法就是两级FF进行同步，也就是我们常说的 “打两拍” ：

        它的时序如下：

        从上图可以看到，两级FF之所以能解决CDC问题的关键在于亚稳态可以在一定时间内自行恢复，这个时间通常在2个时钟周期以内。

        当然了，并不是说亚稳态在第1个FF就一定不能稳定，也不是说亚稳态在第2个FF就一定会稳定。说到底都是个概率问题，理论上你甚至可以打3拍，打4拍，打5拍打10086拍等，这样亚稳态发生的概率会更低，只是打两拍，已经可以把亚稳态发生的概率降低到一个在工程上可以接受的范围内了。

        亚稳态只能被抑制，不能被彻底消除。在一般的FPGA设计中，2级FF同步基本已经够用了。如果你的设计频率非常高，那么也可能需要用到3级FF来同步。

2.1.1、亚稳态恢复后的随机值是否会导致功能错误

        有个经常困扰初学者的问题：既然亚稳态恢复后的值是一个随机值（0或者1），那么如果亚稳态恢复到和预期值不一样的值，岂不是后级电路就采集到了错误值，从而引起功能错误？

        首先我们假设同步一个电平信号，如果同步后的值是正确的即1，那么此时就在两个周期后采集到了正确值，对功能无影响，无非就是该信号传递到后级电路的时间慢了一些（2个周期的latency）；如果同步后的值是错误的即0，那么下一个周期不是亚稳态，此时一定能采集到正确值，也就是说正确的值会在3个时钟后才同步到后级电路，同样的，这种情况也对后级电路没有功能上的影响，只是信号传递慢了一些。

        打两拍的做法并不是唯一解，实际应用还是要看自己的设计要求和器件情况来具体分析。除此之外，还有很多做法来解决亚稳态问题，但它们都是在打2拍的基础上的延伸。

  16. )
  17. (
  18. //源时钟域的信号（即同步前的信号）
  19. 	input		clk_source			,	//源时钟（慢时钟）
  20. 	input		rst_source			,	//高有效的同步复位
  21. 	input		sig_level_source	,	//同步前的电平信号
  22.  
  23. //目的时钟域的信号（即同步后的信号）		
  24. 	input		clk_dest			,	//目的时钟（快时钟）
  25. 	input		rst_dest			,	//高有效的同步复位
  26. 	output		sig_level_dest			//同步后的电平信号
  27. );
  28.  
  29. //定义reg
  30. reg	[SYNC_STAGE - 1 : 0]	sig_level_ff		;		//同步寄存器，同步级数取决于SYNC_STAGE
  31. reg							sig_level_source_d1	;		//把原始信号在源时钟打一拍，保证同步前的是时序逻辑而不是组合逻辑
  32.  
  33. assign sig_level_dest = sig_level_ff[SYNC_STAGE - 1];	//同步后的电平信号
  34.  
  35. //把原始信号在源时钟打一拍，保证同步前的是时序逻辑而不是组合逻辑，因为组合逻辑有毛刺，容易出问题，且会降低MTBF
  36. always @(posedge clk_source)begin
  37. 	if(rst_source)
  38. 		sig_level_source_d1 <= 1'b0;
  39. 	else
  40. 		sig_level_source_d1 <= sig_level_source;
  41. end
  42.  
  43. //把打拍后的信号通过多级同步器同步到目的时钟域
  44. always @(posedge clk_dest)begin
  45. 	if(rst_dest)
  46. 		sig_level_ff <= {(SYNC_STAGE){1'b0}};	//这样写不会报警告
  47. 	else
  48. 		sig_level_ff <= {sig_level_ff[SYNC_STAGE-2:0],sig_level_source_d1};
  49. end
  50.  
  51. endmodule

2.1.2、同步器的输入：源时钟寄存

        采用同步器有一个要求：要跨时钟的信号，需要在源时钟域先经过一级FF进行寄存输出。也就是说，跨时钟信号必须是寄存器的输出，中间不能有组合逻辑。如果输入不是寄存器的输出，而是组合逻辑的输出那么情况会是怎样的呢？如下图所示：

        可以看到，adat是组合逻辑的输出，变化的频率更快了，根据MTBF公式：

        数据变化太快，会减小MTBF，增加亚稳态发生的概率。如果adat是FF的输出，那么根据FF的特性，输出在一个时钟周期内是不会改变的，数据的变化频率不会超过时钟频率，这样就能降低跨时钟信号变化的频率，减小亚稳态发生的概率，如下图所示：

        所以，在使用同步器同步信号时，要求输入信号必须是源时钟域的寄存输出。这一点是很容易犯错和很多老手容易忘记的，只有理解了其中的原理，才能明白为什么要这么做，才会在使用的时候不会犯错。

2.2、脉冲信号的跨时钟域处理（脉冲同步器）

        跨时钟域处理首先要解决的是能不能采集到的问题，然后才是怎么采的问题。因为脉冲信号存在的时间很短，所以如果目的时钟比源时钟要慢，那么是有可能采集不到信号的。

        比如下面的脉冲信号就无法被采集到，这个时候就会漏采而导致后级电路出现功能错误：

        根据源时钟和目的时钟的频率快慢关系，可以把脉冲信号的跨时钟域同步情况分为两类：

  • 从慢时钟域同步到快时钟域

  • 从快时钟域同步到慢时钟域

2.2.1、从慢时钟域同步到快时钟域

        首先需要约定的是，快时钟频率应该至少是慢时钟频率的两倍。若无法满足，则可使用下一节的方法进行处理。在这一前提条件下，是可以保证需要处理的信号能被目的时钟域正确采集到的。

        在实践中，通常只要保证1个目的时钟周期内有3个源时钟域的边沿（上升沿和下降沿都可以），即可保证一定能采集到该信号，即所谓的 “三边原则”。

        在保证能采集到信号的情况下，就可以采用和电平同步器一样的方法来进行同步。不同的是，因为被同步的是脉冲信号，那我们自然也需要同步后的信号是一个脉冲信号，如果直接套用上面的方法，当目的时钟比较快时则有可能出现如下的情况：

        

        因为目的时钟比较快，所以采集了3次，原本的脉冲信号变成了3个周宽度的电平信号。解决办法很简单，只要用取边沿的方法即可。需要注意的是，这里是用的打2拍和打3拍的信号来提取上升沿，原因之前说过打1拍的信号是不稳定的，可能存在亚稳态。当然了，你也可以把这个信号打一拍，这样时序更优，但是相应的也会多一个周期的延迟。

assign sig_ff = sig_d2 && ~sig_d3

        对应的Verilog代码：

  1.  
  2. // ****************************************************************************************************************
  3. // ** 作者 ： 孤独的单刀                                                   			
  4. // ** 邮箱 ： zachary_wu93@163.com
  5. // ** 博客 ： https://blog.csdn.net/wuzhikaidetb 
  6. // ** 日期 ： 2024/04/24
  7. // ** 模块 ： 脉冲信号的CDC同步模块	
  8. // ** 功能 ： 1、将脉冲信号从慢时钟域同步到快时钟域
  9. //			  2、使用时需要保证快时钟域（目的时钟）频率至少是慢时钟域（源时钟）的1.5倍以上，才能保证不漏采
  10. //			  3、同步级数参数化可配置
  11. //
  12. // ****************************************************************************************************************	
  13.  
  14. module sync_pulse #(
  15. 	parameter	SYNC_STAGE = 3			//同步级数，一般配置为2级即可，若频率较高则可配置为3级
  16. )
  17. (
  18. //源时钟域的信号（即同步前的信号）
  19. 	input		clk_source			,	//源时钟（慢时钟）
  20. 	input		rst_source			,	//高有效的同步复位
  21. 	input		sig_pulse_source	,	//同步前的脉冲信号
  22.  
  23. //目的时钟域的信号（即同步后的信号）		
  24. 	input		clk_dest			,	//目的时钟（快时钟）
  25. 	input		rst_dest			,	//高有效的同步复位
  26. 	output	reg	sig_pulse_dest			//同步后的脉冲信号
  27. );
  28.  
  29. //定义reg
  30. reg	[SYNC_STAGE : 0]	sig_pulse_ff		;	//同步寄存器，同步级数要加1，因为第1拍是不稳定的，为了提取上升沿只能用第2拍和第3拍
  31. reg						sig_pulse_source_d1	;	//把原始信号在源时钟打1拍，保证同步前的是时序逻辑而不是组合逻辑
  32.  
  33.  
  34. //把原始信号在源时钟打一拍，保证同步前的是时序逻辑而不是组合逻辑，因为组合逻辑有毛刺，容易出问题，且会降低MTBF
  35. always @(posedge clk_source)begin
  36. 	if(rst_source)
  37. 		sig_pulse_source_d1 <= 1'b0;
  38. 	else
  39. 		sig_pulse_source_d1 <= sig_pulse_source;
  40. end
  41.  
  42. //把打拍后的信号通过多级同步器同步到目的时钟域
  43. always @(posedge clk_dest)begin
  44. 	if(rst_dest)
  45.         sig_pulse_ff <= {(SYNC_STAGE+1){1'b0}};	//这样写不会报警告
  46. 	else
  47. 		sig_pulse_ff <= {sig_pulse_ff[SYNC_STAGE-1:0],sig_pulse_source_d1};
  48. end
  49.  
  50. //取上升沿作为同步后的脉冲信号，这里最好可以用时序逻辑
  51. always @(posedge clk_dest)begin
  52. 	if(rst_dest)
  53. 		sig_pulse_dest <= 1'b0;
  54. 	else
  55. 		sig_pulse_dest <= ~sig_pulse_ff[SYNC_STAGE] && sig_pulse_ff[SYNC_STAGE-1];	//提取上升沿
  56. end
  57.  
  58. endmodule

  1.  
  2. `timescale 1ns / 1ns		//时间单位/精度
  3.

2.2.2、从快时钟域到慢时钟域（握手同步/展宽）

        信号从快时钟域同步到慢时钟域是有很大可能采集不到的，比如下图，adat是快时钟域a下的脉冲信号，bclk是目的时钟（慢时钟），很明显，因为信号adat还没有快时钟域的一个时钟周期宽，所以在被同步到目的时钟域后出现了漏采。

        既然脉冲信号频率高于目的时钟才导致的漏采，那么我将脉冲信号拓宽到一定的程度不就可以保证采集到了吗？这一方法的本质实质上是降频，也就是将问题转换为我们已经能解决的用较快的时钟来采集较慢的时钟。如下图：

        将信号adat扩宽为3倍aclk后，脉冲信号成为了一个较慢的信号，而目的时钟成为了一个较快的时钟，这样我们就可以使用2级同步器的方法来对被CDC信号进行同步。

        需要注意的是，这一方法（开环法）具有一定的局限性：

  1. 若需要同步的两个脉冲信号距离很近，则第一个脉冲信号的扩宽可能会覆盖掉第二个脉冲信号，从而导致第二个脉冲信号的漏采；

  2. 脉冲信号的扩宽是以目的时钟为参考的，但是若不知道目的时钟的频率，则该方法可能失效，也就是说该方法不具备普遍性。

        为了探寻一种具备普遍性的方法，我们可以使用握手法（闭环法）来进行CDC。握手法的本质是负反馈，通俗来讲，就是我先将被CDC信号展宽，展宽后将其同步到目的时钟域，在目的时钟域生成指示信号，该指示信号用来指示此时信号已经被目的时钟域接收，然后将指示信号反馈到源时钟域（反馈过程），源时钟域接收到这个反馈信号后将被CDC信号拉低，从而确定了展宽长度，通过 ”发送--反馈--操作“ 这一握手过程完成了一次CDC传输。

        上图是通过典型的握手来实现CDC：

  1. 源时钟域aclk下的信号adt1信号是要进行CDC的信号；

  2. adt1先是在源时钟域aclk下被展宽，然后通过两级同步器被同步到目的时钟域bclk下，分别为bq1_dat，bq2_dat；

  3. bq2_dat作为指示信号（反馈信号，也可以通过bq1_dat和bq2_dat来生成新的指示信号）又被反馈到了目的时钟域aclk下，并进行同步，分别为aq1_dat，aq2_dat；

  4. aq2_dat的拉高则说明反馈信号的同步完成，此时可以将adt1拉低（结束展宽过程）；

  5. adt1拉低（结束展宽过程）后表示一次CDC操作结束。

        根据上面的波形图很简单就可以写出握手法的Verilog代码：

  1.  
  2. // ****************************************************************************************************************
  3. // ** 作者 ： 孤独的单刀                                                   			
  4. // ** 邮箱 ： zachary_wu93@163.com
  5. // ** 博客 ： https://blog.csdn.net/wuzhikaidetb 
  6. // ** 日期 ： 2024/04/24
  7. // ** 模块 ： 脉冲信号的CDC同步模块	
  8. // ** 功能 ： 1、将脉冲信号完成跨时钟域同步
  9. //			  2、适用从慢时钟域同步到快时钟域，也适用从快时钟域同步到慢时钟域
  10. //			  3、同步级数参数化可配置
  11. //			  4、为了优化逻辑，这里假设要同步的信号是已经是时序逻辑信号了
  12. // ****************************************************************************************************************	
  13.  
  14. module sync_shake #(
  15. 	parameter	SYNC_STAGE = 2			//同步级数，一般配置为2级即可，若频率较高则可配置为3级
  16. )
  17. (
  18. //源时钟域的信号（即同步前的信号）
  19. 	input		clk_source			,	//源时钟（慢时钟）
  20. 	input		rst_source			,	//高有效的同步复位
  21. 	input		sig_pulse_source	,	//同步前的脉冲信号
  22.  
  23. //目的时钟域的信号（即同步后的信号）		
  24. 	input		clk_dest			,	//目的时钟（快时钟）
  25. 	input		rst_dest			,	//高有效的同步复位
  26. 	output		sync_busy			,	//握手忙信号，高电平表示当前正在进行握手过程，此时不应该再发起一次新的请求
  27. 	output	reg	sig_pulse_dest			//同步后的脉冲信号
  28. );
  29.  
  30. //定义reg
  31. reg	[SYNC_STAGE : 0]	sig_pulse_ff			;		//展宽后的信号的同步寄存器，同步级数要加1，因为第1拍是不稳定的，为了提取上升沿只能用第2拍和第3拍
  32. reg	[SYNC_STAGE -1 : 0]	sig_pulse_fback_ff		;		//反馈信号的同步寄存器，同步级级不需要加1
  33. reg						sig_pulse_source_level	;		//打拍后的信号在源时钟域展宽
  34.  
  35. assign sync_busy = sig_pulse_source_level || sig_pulse_fback_ff[SYNC_STAGE -1];	//握手过程
  36.  
  37.  
  38. //在源时钟域，把脉冲信号信号展宽成电平信号，这样可以保证同步后一定能被采集到
  39. always @(posedge clk_source)begin
  40. 	if(rst_source)
  41. 		sig_pulse_source_level <= 1'b0;
  42. 	else if(sig_pulse_fback_ff[SYNC_STAGE -1])		//握手完成后的反馈信号	
  43. 		sig_pulse_source_level <= 1'b0;				//当同步完成后则拉低
  44. 	else if(sig_pulse_source)
  45. 		sig_pulse_source_level <= 1'b1;				//当脉冲有效则拉高
  46. end
  47.  
  48. //在目的时钟域，把展宽后的信号同步过来，为了提取上升沿需要多同步1级，例如设定SYNC_STAGE为2，则应做3个FF同步
  49. always @(posedge clk_dest)begin
  50. 	if(rst_dest)
  51. 		sig_pulse_ff <= {(SYNC_STAGE+1){1'b0}};		//这样写不会报警告
  52. 	else
  53.         sig_pulse_ff <= {sig_pulse_ff[SYNC_STAGE-1:0],sig_pulse_source_level};	//多级FF同步
  54. end
  55.  
  56. //在目的时钟域，提取取上升沿作为同步后的脉冲信号
  57. always @(posedge clk_dest)begin
  58. 	if(rst_dest)
  59. 		sig_pulse_dest <= 1'b0;
  60. 	else
  61.         sig_pulse_dest <= ~sig_pulse_ff[SYNC_STAGE] && sig_pulse_ff[SYNC_STAGE-1];	//上升沿
  62. end
  63.  
  64. //把同步到目的时钟域的展宽信号再反馈回去，这样可以将源时钟域的展宽信号拉低，结束一次握手进程
  65. always @(posedge clk_source)begin
  66. 	if(rst_source)
  67. 		sig_pulse_fback_ff <= {(SYNC_STAGE){1'b0}};		//这样写不会报警告
  68. 	else							
  69. 		sig_pulse_fback_ff <= {sig_pulse_fback_ff[SYNC_STAGE-1:0],sig_pulse_ff[SYNC_STAGE-1]};	//多级FF同步
  70. end
  71.  
  72. endmodule

        仿真结果如下：

        从上图可以看出：

  • 设计了一个sync_busy作为握手过程的指示，当其为高时，说明正在进行一次握手，若发起新的脉冲则会被忽略掉

  • 在源时钟域一共发起了7个脉冲，但是有4个脉冲发起的时间刚好处于上一个脉冲的握手周期内（sync_busy有效期间），所以这4个脉冲被漏掉了，从这也可以看出握手法最大的缺点就是花费的时间长，如果两次脉冲间隔很近，那么后面的脉冲是有可能被漏掉的

  • 握手过程如下：

    1. 从脉冲信号展宽到电平信号，这样可以确保在目的时钟域一定会采到

    2. 展宽后的电平信号信号同步到目的时钟域，第1个FF不稳定可能有亚稳态，所以不能使用

    3. 展宽后的电平信号信号同步到目的时钟域，第2个FF稳定，大概率没有亚稳态，所以可以使用，作为反馈信号反馈到源时钟域，表明握手成功，可以把展宽的电平信号拉低了

    4. 反馈信号同步到源时钟域的第1级，不稳定不能使用

    5. 反馈信号同步到源时钟域的第2级，稳定可以使用

    6. 源时钟域接收到这个信号后就可以拉低展宽信号了

        因为有了busy信号，实际上我们是可以利用它来反压前级的，这样可以有效地防止请求信号的丢失，把TB改一下，如果busy有效就不生成新的脉冲：

sig_pulse_source <= $random && (~sync_busy);

        

        仿真结果，只生成了3个脉冲，都是在握手不忙期间生成的，这样就不会丢数据。如果你的脉冲一定根据需求是一定会间隔这么近的话，那么握手法其实是不适用的，更合适的方法是异步FIFO。
