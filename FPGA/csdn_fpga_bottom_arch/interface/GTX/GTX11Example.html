1、Example Design的架构

示例工程的架构如下：

主要由3个部分组成，封装好的GT IP核，数据生成模块和数据校验模块。层次级如下：

  • 1、顶层模块，主要是对子层模块的封装和连接

  • 2、实现GT IP核的主要模块

  • 3、生成了GT所需要的4个时钟

  • 4、MMCM，作用就是生成3要的时钟

  • 5、QPLL模块原语

  • 6、生成对QPLL模块的同步复位信号

  • 7、数据生成模块

  • 8、数据校验模块

2、顶层模块

顶层模块的示意图如下，主要就是起一个连接的作用：

  1. GT的接收差分对，对应的，发送差分对在⑧处

  2. GT BANK的参考时钟（差分时钟）

  3. DRP的时钟，这里用了IBUFDS从差分转单端

  4. 复位同步模块，即异步复位同步释放，把TX端的复位完成信号作为复位输入到数据发生模块，防止过早地产生数据

  5. 作用同上

  6. 数据生成模块，从ROM里读出数据，然后通过GT发送

  7. 封装后的GT收发器

  8. GT的发送差分对

  9. 数据校验模块

⑦作为被封装的GT模块，内部还有一些细节：

  1. 同步复位模块，把来自外部即用户的软复位同步化后作为QPLL的复位之一

  2. 来自①的复位信号和GT内部的复位信号一起作为QPLL的复位（或门，表示二者都有效）

  3. 这个Quad的QPLL，作用是想GT提供时钟

  4. 时钟生成，TXUSRCLK/TXUSRCLK2 和 RXUSRCLK/RXUSRCLK2 ，都是有TXOUTCLK生成

  5. MMCM来生成所需的时钟

  6. IBUFDS_GTE2，GT参考时钟的专用单端转换模块，将来自外部的参考时钟转换成单端时钟

  7. GT IP核

3、数据生成模块

下面我们来看一下生成测试数据的代码（数据发送端）。 

    initial begin
        $readmemh("gt_rom_init_tx.dat",rom,0,511);
    end

上面的代码很简单就是使用$readmemh函数从 gt_rom_init_tx.dat文件中读出数据，再将数据存放到 512 位深度的 ROM 中。 

always @(posedge USER_CLK) 
    if(system_reset_r2 || (read_counter_i == "111111111")) begin 
        read_counter_i <= `DLY 9'd0; 
    end 
    else 
        read_counter_i <= `DLY read_counter_i + 9'd1;
        上面的代码表示，在 USER_CLK 的时钟上升沿到 来时就将 read_counter_i 信号进行累加，实现了 read_counter_i 从 0 累加到 511 的功能。
always @(posedge USER_CLK) 
     tx_data_ram_r <= `DLY rom[read_counter_i]; 
        上面的代码就是从 ROM 中读出数据，最后将数据送到 TX 端的输出 端口上进行数据发送。

4、数据校验模块

对生成测试数据的代码（数据发送端）的介绍就到这里， 下面我们来介绍检测接收数据部分（数据接收端）。在接收数据检测的代码中，我们可以看出 gtwizard_0_GT_FRAME_CHECK 模块设置了一个参数 START_OF_PACKET_CHAR，代码如下所示： 

module gtwizard_0_GT_FRAME_CHECK # 
( 
     // parameter to set the number of words in the BRAM 
     parameter RX_DATA_WIDTH = 64, 
     parameter RXCTRL_WIDTH = 2, 
     parameter WORDS_IN_BRAM = 512, 
     parameter CHANBOND_SEQ_LEN = 1, 
     parameter COMMA_DOUBLE = 16'hf628, 
     parameter START_OF_PACKET_CHAR = 64'h00000000000000fb 
)

在 GTX IP 的设置时，有个选项叫做 “Align to ” 对齐边界，在检测接收数据的代码里将接收的数据与和设定的参数 START_OF_PACKET_CHAR 进行对比，判断 comma 码出现在接收数据（32 位 位宽，四个字节）的第几个字节，代码如下所示： 

    always @(posedge USER_CLK)
    begin
        if(reset_on_error_in_r2 || system_reset_r2)    sel <= 2'b00;
        else if (begin_r && !rx_chanbond_seq_r)
        begin
            // if Comma appears on BYTE3 ..
            if((rx_data_r[(RX_DATA_WIDTH - 1) : 3*RX_DATA_WIDTH/4] == START_OF_PACKET_CHAR[7:0]) && rxctrl_r[3])
                sel <= 2'b11;        
            // if Comma appears on BYTE2 ..
            else if((rx_data_r[(3*RX_DATA_WIDTH/4 - 1):2*RX_DATA_WIDTH/4] == START_OF_PACKET_CHAR[7:0]) && rxctrl_r[2])
            begin
                sel <= 2'b10;
            end
            // if Comma appears on BYTE1 ..
            else if((rx_data_r[(2*RX_DATA_WIDTH/4 - 1):RX_DATA_WIDTH/4] == START_OF_PACKET_CHAR[7:0]) && rxctrl_r[1])
            begin
                sel <= 2'b01;
            end
            // if Comma appears on BYTE0 ..
            else if((rx_data_r[(RX_DATA_WIDTH/4 - 1):0] == START_OF_PACKET_CHAR[7:0]) && rxctrl_r[0])
            begin
                sel <= 2'b00;
            end
        end      
    end

为了检测数据在发送的过程中是否出现错位现象，需要将 comma 码放在接收数据中同一个字节中，发送模块将 comma 码放了在最低字节（K28.5即0xbc）。如果 comma 码没有放在最低字节，就对接收的数据进行数据右移处理，经过右移之后 comma 码就会放在最低字节，代码如下： 

    always @(posedge USER_CLK)
    begin
        if(system_reset_r2)    rx_data_r3 <= 'h0;
        else 
        begin
            if(sel == 2'b01)
            begin
                rx_data_r3   <=  `DLY    {rx_data_r[(RX_DATA_WIDTH/4-1):0],rx_data_r2[(RX_DATA_WIDTH - 1):RX_DATA_WIDTH/4]};  
            end
            else if(sel == 2'b10)
            begin
                rx_data_r3   <=  `DLY    {rx_data_r[(2*RX_DATA_WIDTH/4-1):0],rx_data_r2[(RX_DATA_WIDTH - 1):2*RX_DATA_WIDTH/4]};  
            end
            else if(sel == 2'b11)
            begin
                rx_data_r3   <=  `DLY    {rx_data_r[(3*RX_DATA_WIDTH/4 - 1):0],rx_data_r2[(RX_DATA_WIDTH-1):3*RX_DATA_WIDTH/4]};  
            end
            else rx_data_r3  <=  `DLY    rx_data_r2;
        end
    end

如果不做数据对齐，会是什么情况呢？这是GT发送端的数据：

框住的这个数据是约定好的同步码，拿来对齐用的；txcharisk_in为1表示最低字节是K码，0xbc就是K28.5。接收端检测到K28.5后就能在0/1的数字码流中找到字节的边界，但是因为接收的数据是4个字节，所以这个0xbc有可能出现在任意一个字节，所以就需要约定同步码来对齐。接收端波形如下：

rxcharisk_out为4表示次高字节是K码，说明在中间的时候0/1码流对上了。可以发现04bc0302的前一个数据是08070605，在发送端060504bc后跟的是0a090807，那么就算K码在中间被对齐了，那后面数据开始两位也应该是0a09才对啊，怎么全乱套了？

gt的发送和接收都是从第字节开始的，也就是说发送头3个数据 03020100 、 060504bc 和0a090807时，依次出现在数据线上的是：

00 01         02 03 bc 04         05 06 07 08         09 0a

在数据中间检测到K码，就会把中间四个数据组成一个数据并排列好，02放到LSB，04放到MSB，也就是04bc032。同理，后面的数据就是08070605。
